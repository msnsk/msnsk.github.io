ブロックのシーンを作る#

ブロックを画面に複数配置していくことを考えると、毎回メインシーンにブロック用の子ノードを追加するのは効率的ではありませんし、ブロックのノードに仕様変更の必要が生じた場合、一つ一つのブロックを修正するのは大変です。

このように、同じ（または似たような）ノードが複数必要な場合は大まかに以下の2ステップで進めると、開発がとても効率的になります。

  1. 雛形となるシーンを作る
  2. 別のシーンに必要なだけ雛形のシーンのインスタンスを作る

特に修正が必要な場合、雛形のシーンを修正することによって、自動的にそのインタンスにも同様の変更が加えられるので、メンテナンスのしやすさが断然違ってきます。

ということで、まずは雛形となるブロックのシーンを作っていきます。「シーン」メニュー＞「新規シーン」を選択します。

シーンドックで「その他のノード」を選択します。

壁を作った時と同様に「StaticBody2D」を選択して「作成」をクリックします。

「StaticBody2D」ノードが追加できました。これがルートノードになります。名前を「Brick」に変更しましょう。

一旦、このシーンを保存しておきましょう。「シーン」メニュー＞「シーンを保存」を選択するか、ショートカットキー操作でシーンを保存しましょう（シーンを保存：Windows: Ctrl + S / macOS: Cmd + S）。

保存先のパスに「res://scene」を選択したら、ファイル名はそのまま「Brick.tscn」として保存しましょう。

続けて、シーンドックで「Brick」ノードを選択した状態で「Sprite」を子ノードに追加しましょう。

「Brick」ノードに「Sprite」ノードが追加されました。そのまま「Sprite」ノードを選択した状態にします。

インスペクタの「Texture」プロパティめがけて、ファイルシステムドックから「Brick.png」をドラッグ&ドロップします。

「Texture」プロパティにブロック用に用意しておいた画像が反映されました。

続けて「Brick」ノードに「CollisionShape2D」ノードを追加します。

追加されました。そのまま「CollisionShape2D」を選択した状態でにします。

インスペクタの「Shape」プロパティの [空] をクリックして「新規 RectangleShape」を選択して設定します。

それでは「CollisionShape2D」の形を「Sprite」ノードの Texture の形に合わせていきます。2D ワークスペースで作業しますが、先にツールバーのピクセルスナップを有効にし、グリッドスナップを無効にします。これで1ピクセすごとにスナップされます。

赤いポイントをドラッグしながら「Sprite」ノードに合わせます。


ブロックの色を編集しやすくする#

ブロックにスクリプトをアタッチし、コードによって、簡単にブロックの色を編集できるようにしていきます。

スクリプトをアタッチする#

ブロックにスクリプトをアタッチして、必要なプログラミングを実施していきます。まずは「Brick」ノードにスクリプトをアタッチしましょう。

スクリプトのファイル保存先に「res://scripts」を指定して、ファイル名を「Brick.gd」とし、作成しましょう。

スクリプトワークスペースが開き、デフォルトのスクリプトが表示されましたね。ここからこのスクリプトを更新していきます。

今回、スクリプトに記述する全体のコードは以下の通りです。その内容としては、ブロックの色を手軽に編集するためのコードです。コーディングはちょっと苦手、という方はひとまずこのコードでスクリプトのコードを置き換えてください。

tool
extends StaticBody2D


export (Color) var brick_color = Color(1, 1, 1) setget set_color

func _ready():
	set_color(brick_color)

func set_color(color):
	brick_color = color
	if is_inside_tree():
		get_node("Sprite").set_modulate(color)

スクリプト解説#

それでは Brick.gd のスクリプトの内容を小分けにして解説していきます。

tool
extends StaticBody2D

スクリプトは通常、エディタ上では動きませんが、toolキーワードはスクリプトをエディタ上でも動くようにするためのキーワードです。

なぜこれが必要かというと、最終的に、エディタ（インスペクタ）上で「Brick」ノードの「color」というプロパティの値を編集した時に、スクリプトによって自動的に子ノードである「Sprite」ノードの色も変更させて、画面上に表示されるブロックの色を調整できるようにしたいからです。あくまで画面上に表示されるのは「Sprite」ノードなので、「Brick」ノードのプロパティをいじっただけでは画面上のブロックの色は変化しません。

extends キーワードは、すでにご存知の通り、StaticBody2D クラスを拡張することを宣言するために記述されています。

export (Color) var brick_color = Color(1, 1, 1) setget set_color

brick_colorというプロパティを定義しつつ、exportキーワードを頭につけることでインスペクタで編集できるようにしています。このプロパティのデータ型をColorとし、デフォルトの色をColor(1, 1, 1)（白）で指定しています。

setgetキーワードは、セッターおよびゲッターとしてどのメソッドを指定するかを示すキーワードです。このスクリプトでは、セッターとしてset_colorというメソッドを指定しています。

セッターとしてメソッドを指定している場合、そのセッターが紐づいているプロパティの値が外部から変更されたら、そのプロパティの値が変更されるより前に直ちにセッターのメソッドが実行されます。そのメソッドの内容は、そのプロパティに新しい値を代入することが基本的な役割ですが、同時に他の作業も実施させることができます。

具体的には、brick_colorというプロパティの値が、インスペクタ上で変更されようとしたら、そのタイミングで自動的にset_colorというメソッドによってbrick_colorに新しい値が代入されるということです。set_colorメソッドはプロパティに新しい値を代入すると同時に別の作業も実行しますが、その具体的な内容については後ほど解説します。

一方、今回のスクリプトでは使用していませんが、ゲッターとしてメソッドが指定されている場合は、そのゲッターが紐づいているプロパティにアクセスされたら、ゲッターメソッドによって何らかの値が返されます。ただし、ゲッターとしてメソッドを指定する場合はvar xxxxx = x setget ,get_methodのように ,（カンマ） が必要なので要注意です。

次は最後まで一気に見ていきます。

func _ready():
	set_color(brick_color)

_ready()メソッドのブロックの中でset_color()メソッドが実行されています。set_color()メソッドの引数には、先に宣言しておいたプロパティbrick_colorを指定しています。このset_color()というメソッドは、先に説明したbrick_colorプロパティのセッターにもなっていましたね。このメソッドがちょうどこのあとに定義されていますので見ていきましょう。

func set_color(color):
	brick_color = color
	if is_inside_tree():
		get_node("Sprite").set_modulate(color)

まず定義済みのプロパティbrick_colorに引数colorを代入することで、その値を更新しています。続いて、if 構文による条件分岐です。is_inside_tree()は全てのクラスの親である「Node」クラスにもともと用意されているメソッドで、このスクリプトがアタッチされているノード（ここでは「Brick」ノード）がシーンツリー内に存在するかどうかを Bool 型（true か false のいずれかを値にとるデータ型）の値で返します。このようにifの後ろにすぐ Bool 値を返すメソッドが記述された場合、それは「メソッドの実行結果が true だったら」という意味になります。ちなみに「メソッドの実行結果が false だったら」という意味で記述する場合はif not bool_type_method():という書き方になります。

そして、true だった場合は、if構文のブロック内のコードを実行します。最初に出てくるget_node()は、スクリプトがアタッチされているノード（ここでは「Brick」ノード）から見た子ノードにアクセスするためのメソッドです。ここではget_node("Sprite")と記述していますので、子ノードである「Sprite」にアクセスしています。

その後続けて.set_modulate(color)とありますが、まず.はその前のオブジェクトが持つプロパティやメソッドであることを示しています。つまりset_modulate()メソッドは「Sprite」のメソッドということになります。このメソッドは、Sprite クラスの Texture の色を変更します。set_modulate()の引数にset_color()メソッドの引数であるcolorを指定しています。

スクリプトの内容をまとめましょう。まず、エディタ（インスペクタ）上で「Brick」ノードのbrick_colorプロパティの値（色）を変更しようとすると、「brick_color」プロパティ自体の値として更新されるだけでなく、同時に子ノードである「Sprite」ノードの Texture の色も同じ色に更新します。また_ready()メソッドによって、ゲームが開始される際に全てのノードの読み込みが終わった後、ゲーム開始の準備段階でset_color()メソッドが読み込まれ、引数であるbrick_colorプロパティの値が「Sprite」ノードの Texture の色として適用されますので、「Sprite」ノードの Texture に対して別途、色の設定をする必要は全くありません。

難しい部分もあったかもしれませんが、とにかくこのスクリプトで実現したかったのは、簡単にブロックの色を編集できるようにする、ということだけなのです。


ブロックのグループを作って追加する#

先にも説明した通り、「Brick」シーンを雛形（ブループリントや設計図と言ってもいいですね！）として、「Game」シーンに「Brick」シーンのインスタンスを複数作ることになります。そのような状況で、そららの複数のインスタンスに対して何らかの処理を施す場合、それぞれ個別に処理するよりは、それらを一つのグループにまとめて、そのグループに対して1回処理を施したほうが効率的ですね。ということで、これからブロックのグループを作ります。

シーンドックで「Brick」ノードを選択したまま、エディタ右側のインスペクタドックからノードドックに切り替え、グループタブを開きます。枠内にグループ名を入力して、右隣の「追加」をクリックします。今回は「Bricks」というグループ名にしました。

追加できましたね。今は作ったグループには「Brick」ノードしか含まれていませんが、このノードを雛形にしてインスタンスを大量に作成する際、元の「Brick」ノードがすでにグループに追加されているので、生成されたインスタンスも最初からグループに追加されている形になります。


ボールが当たったらブロックが消えるようにする#

それでは久しぶりに「Game」シーンを開きましょう。「シーン」メニュー＞「シーンを開く」を選択します。ショートカットキー操作でも構いません（シーンを開く Windows: Ctrl + O / macOS: Cmd + O）。

「Game.tscn」を選択して開きます。

これから Godot のシグナルの機能を利用して「ボールが当たったらブロックが消える」という仕組みを作っていきます。シーンドックで「Ball」ノードを選択しましょう。

エディタ右側にあるノードドックのシグナルタブを開きます。あらかじめ様々なシグナルが用意されています。これらはノードの種類によってその数や種類が異なります。今回は RigidBody2D ノードに備わっている「body_entered(body: Node)」シグナルを使います。このシグナルを「Ball」ノードのスクリプトに接続していきます。

そのまま「body_entered(body: Node)」シグナルをダブルクリックするか、右下の「接続」をクリックしてください。

「メソッドにシグナルを接続」パネルが表示されたら、「スクリプトに接続」のリストから「Ball」ノードのを選択します。「受信側メソッド」はスクリプトに接続された後に自動的に生成されるメソッドの名前です。メソッドの名前はデフォルトで「on_ノード名_シグナル名」になります。問題なければ「接続」をクリックします。

自動的にスクリプトワークスペースで「Ball.gd」スクリプトファイルが開きます。スクリプトの一番最後に_on_Ball_body_entered()メソッドが追加されているのがわかりますね。メソッドが定義されている行の左側にはシグナルに接続していることを示す緑色のアイコンが表示されるので視覚的にもわかりやすいです。

このメソッドの内容に記述したコードが、シグナルが発せられたタイミングで実行されます。今回利用するシグナル「body_entered(body: Node)」の場合は「Ball」ノードにオブジェクトが衝突したら発せられます。メソッドの引数bodyは「Ball」ノードに衝突したオブジェクトを指します。

結果的に「Ball.gd」のスクリプト全体は以下のようになっています。

extends RigidBody2D


export (float) var ball_speed = 150.0

func _ready():
	var direction = Vector2(1, -1).normalized()
	var velocity = direction * ball_speed
	apply_impulse(Vector2.ZERO, velocity)


func _on_Ball_body_entered(body):
	if body.is_in_group("Bricks"):
		body.queue_free()

それでは_on_Ball_body_entered()メソッドの中身を編集していきましょう。まず以下の1行は削除します。

pass # Replace with function body.

替わりに、以下のコードを記述します。

	if body.is_in_group("Bricks"):
		body.queue_free()

ここの if 構文全体の意味としては『もしbodyが「Bricks」グループに追加されていればbodyを削除する』という条件分岐のコードを記述しています。if 構文のbodyはボールが衝突しうるオブジェクトを指します。具体的には「Paddle」ノード、「Wall」ノード、そしてこれから作る予定の「Brick」シーンのインスタンスノードです。そこで、先に作っておいたノードグループ「Bricks」を利用して、『bodyが「Bricks」グループのメンバーだったら…』という意味のコードにしたのがif body.is_in_group("Bricks"):の部分になります。is_in_group()メソッドは引数のグループに属しているかどうかを Bool 値で返します。

そして、is_in_group("Bricks")メソッドが true の結果を返した場合（bodyが「Brick」のインスタンスだった場合）に実行したいのは「その衝突したブロックを消す」という意味のコードです。queue_free()というのがまさにノード自体をシーンから削除するメソッドですので、これを実行しています。

コードの編集ができたら、「Ball」ノードを選択して、インスペクタ上で、以下の2つのプロパティの値を変更しましょう。

  • Contacts Reported：1 に変更
  • Contact Monitor: オンにする

レベル（ステージ）のシーンを作る#

一般的なブロック崩しゲームは、簡単なレベル1から始まり、段々と難易度が上がっていき、最後のレベルが最も難しいというのがオーソドックスなゲームデザインですね。その難易度を決めるのは、ほとんどがブロックの配置や種類（例えば2回当てないと消せないブロックなど）になります。まずは一つ目のレベルとして最も簡単なレベル1のシーンを作っていきます。


新規シーンを作成する#

「シーン」メニュー＞「新規シーン」を選択します。

「2D シーン」を選択します。これにより、自動的に新規「Node2D」ノードがルートノードとして追加されます。

まず一つ目のレベルシーンを作りますので、「Node2D」ノードの名前を「Level1」に変更しましょう。


「Level1」ノードに「Brick」シーンのインスタンスを追加する#

「Level1」ノードの子ノードとして、作成済みの「Brick」シーンをインスタンス化して追加します。手順は次のどちらでも構いません。

＜手順その1＞#

シーンドック上で「Level1」シーンを右クリックして、「子シーンをインスタンス化」を選択します。

インスタンス化したいシーンのファイルを選択します。ここでは「scene/Brick.tscn」を選択して「開く」をクリックします。

＜手順その2＞#

ファイルシステムドックから、インスタンス化したいシーンのファイルを、シーンドックの親ノード（ここでは「Level1」ノード）にドラッグ&ドロップします。

いずれかの手順によって、「Brick」シーンのインスタンスが「Level1」ノードの子ノードとして追加されましたね。

この後、このノードを量産していく作業をわかりやすくするために、ノードの名前を「Brick1」に変更します。

ちなみに、
は何らかのノードグループに属していることを示しています。
は別のシーンのインスタンスであることを示しています。

ここで一度作成した「Level1」のシーンを保存しておきましょう。保存先パスは「res://scene」、ファイル名は「Level1.tscn」で問題ありません。


「Brick」ノードを量産して配置する#

それでは「Brick1」ノードを複製してそれを移動させる、という作業を繰り返していくための下準備をします。まず、ブロックの色を行によって変えることにします。インスペクタで、1行目の1つ目のブロックとなる「Brick1」ノードの「Brick Color」プロパティを選択してカラーピッカーで色を選択（または入力）してください。

色が更新されましたね。

次に 2D ワークスペースのグリッドの間隔を調整します。ツールバーの「スナッピングオプション」をクリックして「スナップの設定」をクリックします。

グリッドのステップを「Brick1」ノードのサイズと同じ横32px、縦16pxに変更して「OK」をクリックします。

ツールバーで「移動モード」を有効にし（移動モード Windows: W / macOS W）、グリッドスナップも有効にします。

座標(0, 0)にある「Brick1」ノードを縦に3グリッド分、横に3グリッド分ドラッグして移動させましょう。

positionプロパティの値は(96, 48)になっています。

ここからは簡単です。以下の操作を10回繰り返して横にブロックが11個並んだ状態を作ります。

  1. シーンツリー上一番下にあるブロックのノード（最初は「Brick1」）を選択する
  2. ショートカットキー（Windows: Ctrl + D / macOS: Cmd + D）で選択したブロックのノードを複製
  3. 複製したブロックを 2D ワークスペース上で横に1グリッド分移動

それではシーンを実行して確認していきましょう。ただし、色も確認したいので、先に「デバッグ」メニュー＞「コリジョン形上の表示」のチェックを外しておきます。チェックが入りだと、コリジョン形上を表す半透明の緑色が重なってしまうので、実際のブロックの色を確認できないからです。

シーンを実行します。

自分で指定した色のブロックが横一列に並んでいますね。

先のブロック複製作業をさらに繰り返して、画面上に4段ほどブロックを並べましょう。ゲーム性には直接関係ありませんが、見た目を少し楽しくするために一段ずつ色を変えます。

それでは再度シーンを実行して、ブロックの配置を確認しましょう。


レベルシーンをゲームシーンの子ノードにする#

最初のレベルのシーンである「Level1」が完成したので、これをインスタンス化して「Game」シーンに子ノードとして追加します。

「Game」シーンに切り替えたら、「Level1.tscn」のインスタンスを「Game」ノードの子ノードとして追加します。

ようやく 2D ワークスペースにブロック崩しに必要な全てのノードが揃いましたね。

最後にプロジェクトを実行して遊んでみましょう。

若干パドルのスピードが遅くてハードモードですが、ボールがブロックに当たった時はブロックが消えて、パドルと壁に当たった時は跳ね返りますね。
